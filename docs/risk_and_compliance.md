# 风险与合规边界

## 医疗边界

Hospital Guide Agent 当前仅用于医院导诊与挂号辅助原型展示。系统可以根据结构化规则和 Mock Data 提供科室推荐、急诊入口提示、院内位置指引和排班/号源查询，但不能替代医护人员判断。

系统通过有限多轮澄清收集导诊所需信息，并在候选规则和结构化知识库范围内推荐科室或急诊入口；该过程不用于诊断疾病，也不用于治疗或用药建议。

本项目只做：

- 科室推荐
- 急诊入口提示
- 院内位置指引
- 挂号 / 排班辅助
- 信息不足时的有限追问
- 红旗表现识别与就诊入口路由

## 禁止能力

本项目不做：

- 诊断
- 治疗
- 用药建议
- 评估病情轻重
- 检查、手术、处方建议
- 对真实病历、检查报告或影像结果作医学结论
- 对用户是否需要住院、手术或特定处置作判断

## 安全控制机制

当前原型包含以下安全控制：

- 系统提示词约束：所有 LLM 模块都明确限制为导诊任务。
- LangGraph 状态机：流程由代码控制，LLM 不直接决定系统行为。
- 候选规则内裁决：LLM 只能从候选规则中选择，不得创造科室。
- Triage Interview Planning：仅用于有限多轮澄清、症状信息补齐、候选科室收敛和急诊入口路由，职责是决定继续追问还是进入科室推荐。
- 结构化上下文回复：最终回复只使用科室、位置、排班等结构化字段。
- 信息不足时追问：只追问年龄、性别、主要症状、部位、发作方式和伴随表现。
- 敏感表达后处理：诊断、治疗、用药和评估病情轻重等表达会被拦截。
- 急诊入口提示限制：命中急诊入口时不输出普通号源和预约建议。

## 真实落地前所需审核

进入真实医院环境前，至少需要完成以下审核：

- 医疗规则审核：由对应专科、门诊部或医务部门确认分诊规则和话术。
- 法务合规审核：明确系统定位、免责声明、用户告知和责任边界。
- 信息安全审核：确认接口权限、网络边界、密钥管理和日志留存策略。
- 隐私保护审核：确认个人信息最小化采集、脱敏、访问控制和删除机制。
- 运营流程审核：确认人工接管、异常升级、投诉处理和版本发布流程。
- 评测验收审核：确认测试集、指标阈值、失败样本复盘和上线门槛。

## 隐私说明

当前 Mock Demo 不需要真实患者身份信息。真实落地时应遵循最小必要原则：

- 不采集与导诊无关的身份信息。
- 日志中避免保存完整身份证号、手机号、医保号等敏感字段。
- 对用于模型评测和规则优化的数据进行脱敏。
- 对日志访问设置权限、审计和留存期限。
- 明确告知用户系统用途和能力边界。

## 日志与审计

真实系统建议记录以下结构化日志：

- 用户意图类别
- 当前阶段
- 已抽取槽位
- 命中的规则 ID 或候选规则列表
- 调用的工具名称和工具结果摘要
- 最终输出版本
- 是否触发安全拦截
- 是否转人工

日志不应记录隐藏推理链，也不应保存不必要的敏感原文。

## 人工接管

真实落地时需要设计明确的人工接管机制：

- 用户持续要求诊断、治疗、用药或检查建议。
- 用户表达紧急或复杂情况，但规则无法可靠路由。
- 系统连续追问后仍无法获得必要信息。
- 位置、排班或接口结果缺失。
- 用户主动要求人工服务。

## 规则审核

分诊规则和话术应采用版本化管理：

- 每条规则包含适用范围、推荐科室、急诊入口标记和审核状态。
- 规则变更需要记录审核人、审核时间和变更说明。
- 评测集应覆盖新增和修改规则。
- 生产环境发布前应完成回归评测。
