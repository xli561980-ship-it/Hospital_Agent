# 实施路线图

本路线图描述从当前 Mock Demo 到真实医院系统的可行扩展路径。各阶段应以业务验证、安全边界和评测验收为前提逐步推进。

## Phase 0：Mock Demo

目标：

- 展示受控型医院导诊 Agent 的端到端流程。
- 使用 Mock Data 覆盖分诊规则、院内位置和医生排班。
- 保持本地可运行、可演示、可评测。

接入系统：

- 本地 JSON 知识库
- Streamlit Web Demo
- 离线评测数据集

验收指标：

- Demo 能通过 `streamlit run app.py` 启动。
- 评测脚本能在无 LLM Key 模式下运行。
- 输出明确遵守医疗安全边界。

风险控制：

- 不接入真实患者数据。
- 不声称为生产系统。
- 不输出诊断、治疗、用药或严重程度判断。

## Phase 1：PoC

目标：

- 接入真实科室目录和经过审核的导诊规则。
- 接入部分排班接口或静态排班快照。
- 用小范围业务样本验证导诊准确性和失败模式。

接入系统：

- 科室知识库
- 排班查询接口或排班数据同步表
- 规则审核后台或版本化文件

验收指标：

- 核心科室推荐准确率达到约定阈值。
- 急诊入口提示样本无漏拦截。
- 位置和排班结果与源系统一致。
- 失败样本可复盘到规则、槽位或意图问题。

风险控制：

- 使用脱敏样本。
- 限定试点科室和场景。
- 所有规则和话术经过业务审核。

## Phase 2：Pilot

目标：

- 在有限真实入口中试点，例如医院小程序、公众号、院内屏或客服坐席辅助。
- 接入人工接管和日志审计。
- 验证真实流量下的稳定性、转化率和安全边界。

接入系统：

- 小程序 / 公众号 / 院内屏
- 人工客服系统
- 院内地图或位置服务
- 日志与审计平台

验收指标：

- 用户问题分流率
- 导诊台重复咨询减少比例
- 人工接管触发率和处理时长
- 位置查询命中率
- 安全拦截触发和复核结果

风险控制：

- 灰度发布。
- 高风险表达转人工。
- 日志脱敏和权限控制。
- 每周复盘失败样本并更新规则。

## Phase 3：Production

目标：

- 建设可长期运行的生产系统。
- 接入 HIS、预约挂号、院内地图、统一身份和权限系统。
- 建立持续评测、规则治理、审计报表和应急处置机制。

接入系统：

- HIS / 门诊业务系统
- 预约挂号系统
- 医生排班系统
- 院内地图系统
- 统一认证与权限控制
- 日志审计与告警系统

验收指标：

- 可用性和延迟满足医院入口要求。
- 核心场景准确率和召回率持续达标。
- 安全拦截、转人工和审计链路可追踪。
- 数据权限、隐私脱敏和留存策略通过审核。
- 规则更新有版本、审批、回滚和评测记录。

风险控制：

- 生产变更审批。
- 回滚机制。
- 接口限流和降级方案。
- 安全事件响应流程。
- 定期红队测试和合规复核。
